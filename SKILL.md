# 论文精读 Skill

## 1. 适用场景

当你需要深入理解一篇论文，而不是只看摘要时，使用这个 skill。

适用于：

- 读一篇新论文；
- 准备组会汇报；
- 写论文笔记；
- 判断论文创新性；
- 复现论文方法；
- 比较多篇相关工作；
- 提炼论文中的可迁移思想。

---

# 2. 精读总目标

精读一篇论文时，不要只回答"这篇论文讲了什么"，而要回答下面六个问题：

1. **它研究的任务是什么？**
2. **为什么这个任务重要？**
3. **已有方法为什么不够？**
4. **作者提出了什么新方法？**
5. **实验如何证明这个方法有效？**
6. **这个方法真正贡献在哪里，又有什么局限？**

一句话：

> 精读论文 = 还原作者的问题意识、方法设计逻辑、证据链和适用边界。

---

# 3. 阅读原则

## 原则一：先读问题，再读方法

不要一上来陷入公式、模块名、实验表格。

先搞清楚：

- 论文想解决什么问题；
- 这个问题在什么背景下出现；
- 为什么以前的方法解决不好；
- 作者认为难点在哪里。

如果问题没搞清楚，方法细节读得再多也容易碎片化。

---

## 原则二：把论文读成一条因果链

精读时要把论文整理成下面这条链：

```text
现实背景
  ↓
核心问题
  ↓
已有方法不足
  ↓
作者洞察
  ↓
方法设计
  ↓
实验验证
  ↓
结论与局限
```

每读一节，都要问：
这一节在整篇论文的论证链条里起什么作用？

## 原则三：区分"作者证明了什么"和"作者声称了什么"

论文里的内容可以分成三类：

| 类型 | 含义 | 处理方式 |
| :--- | :--- | :--- |
| 事实 | 数据集规模、实验结果、算法流程 | 精确记录 |
| 论证 | 作者用实验支持的结论 | 看证据是否充分 |
| 推测 | 作者的解释、未来展望、可能影响 | 标注为推测，不当作事实 |

精读时尤其要警惕：
作者说"显著提升"，提升到底多大？
作者说"泛化性好"，实验是否真的覆盖了泛化？
作者说"无需外部反馈"，是否仍然使用了 judge、rubric、self-play 等内部反馈？
作者说"可迁移"，迁移实验是否充分？

---

# 4. 精读步骤

## Step 1：先做 5 分钟快速定位

先读：
- Title；
- Abstract；
- Introduction；
- Figure 1；
- Main results table；
- Conclusion。

目标是快速回答：
1. 这篇论文研究什么？
2. 为什么要研究？
3. 作者提出的方法叫什么？
4. 方法大概怎么做？
5. 结果比 baseline 好在哪里？

不要在第一遍纠结细节。

## Step 2：提炼一句话摘要

读完第一遍后，必须写出一句话版本：

这篇论文提出了 ________，用于解决 ________ 问题， 核心思想是 ________，实验表明 ________。

例如针对刚才那篇论文：
这篇论文提出 Ctx2Skill，用于解决语言模型从复杂上下文中学习新知识并解决任务的困难； 核心思想是通过多代理自博弈，把上下文自动转化为可复用的自然语言技能； 实验表明，该方法能在 CL-bench 上稳定提升 GPT-4.1、GPT-5.1 和 GPT-5.2 的表现。

如果写不出这句话，说明还没真正进入论文主线。

## Step 3：拆任务背景

阅读 Introduction 和 Problem Setting 时，提取：

- 任务名称：
- 任务输入：
- 任务输出：
- 任务难点：
- 现实意义：
- 与已有任务的区别：

重点问：
- 这是一个新任务，还是旧任务的新设定？
- 它和相关概念有什么区别？
- 作者为什么认为这个任务重要？
- 这个任务在真实应用中对应什么场景？

例如：
- 任务名称：Context Learning
- 任务输入：复杂上下文 + 待解决任务
- 任务输出：符合上下文规则、知识、流程和约束的答案
- 任务难点：模型不仅要检索信息，还要从上下文中学习新规则、新流程、新经验规律
- 现实意义：企业文档、产品手册、法规制度、实验数据分析等都需要这种能力

## Step 4：拆论文动机

动机不是简单地说"现有方法不好"，而是要明确：

- 已有方法 A 做了什么？ 它为什么不够？
- 已有方法 B 做了什么？ 它为什么不够？
- 作者的新洞察是什么？

推荐用这个模板：
过去的方法主要依赖 ________。 但在本文任务中，问题是 ________。 因此作者认为，需要一种 ________ 的方法。

例如：
过去的方法通常让模型直接读长上下文并回答，或者用一次性 prompting 生成摘要/规则。 但在 context learning 中，真正困难的是从复杂上下文中提炼可复用、可执行、可迁移的操作性知识。 因此作者认为，需要一种能从上下文自动发现并演化 skill 的方法。

## Step 5：拆核心方法

读 Method 部分时，不要逐句翻译，而要拆成四层。

### 第一层：方法总览

回答：
- 方法叫什么？
- 输入是什么？
- 输出是什么？
- 中间过程是什么？
- 最终怎么用于推理？

例如：
- 方法名：Ctx2Skill
- 输入：复杂上下文
- 输出：Reasoner skill set
- 中间过程：Challenger 出题，Reasoner 解题，Judge 判题，Proposer 诊断，Generator 更新技能
- 最终使用：把生成的 skill 插入 prompt，帮助模型回答真实任务

### 第二层：模块分工

把每个模块整理成表格：

| 模块 | 输入 | 输出 | 作用 |
| :--- | :--- | :--- | :--- |
| Challenger | 上下文 + 出题技能 | 任务和 rubrics | 暴露 Reasoner 的理解盲点 |
| Reasoner | 上下文 + 任务 + 解题技能 | 答案 | 尝试解决任务 |
| Judge | 答案 + rubrics | pass/fail | 提供反馈 |
| Proposer | 成功/失败案例 | 修改建议 | 诊断技能缺口 |
| Generator | 修改建议 + 当前技能 | 新技能 | 生成可复用 skill |

读方法时一定要问：
这个模块为什么存在？如果去掉会怎样？
这能帮助理解消融实验。

### 第三层：算法流程

把论文方法翻译成伪代码：

1. 初始化 Reasoner skill 和 Challenger skill
2. Challenger 根据上下文生成任务和评分 rubrics
3. Reasoner 使用当前 skill 回答任务
4. Judge 根据 rubrics 判定答案是否通过
5. 对失败任务：
    - 更新 Reasoner skill
6. 对成功任务：
    - 更新 Challenger skill
7. 重复多轮
8. 用 Cross-Time Replay 选择最稳健的 Reasoner skill
9. 推理时使用最终 skill

如果一个方法不能被你写成流程，说明你还没真正理解它。

### 第四层：关键设计理由

每篇好论文都有几个关键设计点。精读时要专门提取：

- 设计点 1：为什么这样设计？
- 设计点 2：解决了什么问题？
- 设计点 3：有没有实验验证？

例如这篇论文的关键设计是：

| 设计 | 解决的问题 |
| :--- | :--- |
| Reasoner skill | 让模型把上下文知识显式化、程序化 |
| Challenger skill | 让出题器不断变强，持续暴露漏洞 |
| Proposer / Generator 分离 | 先诊断失败模式，再写成可复用技能 |
| Cross-Time Replay | 避免自博弈后期过拟合 |
|极端任务 hard/easy probe 同时使用 | 兼顾补短板和防退化 |

---

# 5. 精读实验部分

实验部分不要只看最高分，要读出作者的证明结构。

## Step 1：看实验想证明什么

一般论文实验会证明几类问题：
- RQ1：主方法是否比 baseline 好？
- RQ2：各个模块是否必要？
- RQ3：方法在哪些任务上最有效？
- RQ4：生成结果质量如何？
- RQ5：方法是否能迁移？
- RQ6：方法有什么失败案例？

读实验时要把每张表对应到一个研究问题。

## Step 2：拆主实验表格

主实验表格要看四件事：
1. baseline 是谁；
2. 指标是什么；
3. 提升是绝对提升还是相对提升；
4. 哪些子任务提升最大。

例如：
- GPT-4.1 no skills：11.1
- GPT-4.1 + Ctx2Skill：16.5
- 绝对提升：+5.4

不要只写"提升明显"，要写清楚：
- 在哪个模型上提升？
- 在哪个任务类别上提升？
- 相比哪个 baseline 提升？
- 提升多少？

## Step 3：拆消融实验

消融实验的读法是：
- 完整方法表现是多少？
- 去掉某模块后表现是多少？
- 下降多少？
- 说明这个模块有什么作用？

模板：
当去掉 ________ 时，性能从 ________ 降到 ________。 这说明 ________ 对方法有效性很重要，因为它负责 ________。

例如：
当去掉 Challenger skills evolving 时，GPT-4.1 的 overall 从 16.5 降到 13.8。 这说明 Challenger 的进化是关键组件，因为它能不断生成更有挑战性的任务，从而迫使 Reasoner 发现新的技能缺口。

## Step 4：拆 case study

Case study 不是装饰，而是用来解释"为什么方法有效"。

阅读 case study 时要问：
- 无方法时错在哪里？
- 用了方法后修正了什么？
- 这个修正是否对应方法中的某个模块？
- 这个案例能不能代表普遍问题？

例如：
无 skill 时，模型可能漏掉公式、格式要求或边界条件。 有 skill 时，模型会先提取规则，再逐项检查 rubrics。 这说明 skill 的作用不是提供答案，而是提供解题流程和自检机制。

---

# 6. 精读结论与局限

读 Conclusion 和 Limitation 时，要分成三类：

### 论文明确结论

作者确实通过实验支持的结论。
- Ctx2Skill 在 CL-bench 上提升了多个 backbone 的表现。
- Cross-Time Replay 能缓解自博弈后期过拟合。
- 强模型生成的 skill 可以迁移给弱模型使用。

### 合理推论

论文没有直接完全证明，但可以从结果中推导。
- 自然语言 skill 可以作为一种中间知识表示。
- 对于重复使用同一上下文的任务，先生成 skill 再推理可能比每次直接读上下文更有效。

### 局限

需要明确写出：
- 成本高不高？
- 依赖什么模型？
- 评测是否单一？
- 是否可能过拟合 benchmark？
- 是否需要强 judge？
- 是否容易产生错误 skill？

例如：
- Ctx2Skill 需要多轮多代理调用，成本较高。
- 它依赖强 judge 判断 rubrics 是否通过。
- 它主要在 CL-bench 上验证，是否能泛化到其他真实场景还需要更多实验。

# 8. 精读自检清单

精读完成后，用下面清单检查自己是否真正读懂：

- [ ] 我能用一句话讲清楚论文做了什么。
- [ ] 我能解释这个问题为什么重要。
- [ ] 我能说清楚已有方法为什么不够。
- [ ] 我能画出方法流程图。
- [ ] 我能解释每个模块的作用。
- [ ] 我能说明主实验表格证明了什么。
- [ ] 我能解释至少两个消融实验。
- [ ] 我能说出方法最有效的场景。
- [ ] 我能说出方法不适合的场景。
- [ ] 我能区分作者证明的结论和我的推测。
- [ ] 我能提出至少一个批判性问题。
